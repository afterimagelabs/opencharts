# OpenCharts — Agent Context

This is the OpenCharts repo: a public records-request tracking toolkit. It started as a Cloudflare Pages site that read from the CMS (Ezra Legal) Supabase project and rendered a public timeline at `opencharts.org/request/<hash>`. It's being rebuilt as a multi-tenant SaaS.

## ⚠️ Branch convention — read before pushing

There are **two long-lived branches**, each with its own Cloudflare Pages deploy:

| Branch | What's on it | What CF Pages does with it |
|--------|--------------|----------------------------|
| `production-ezra` | The original Ezra-only build. Reads from the CMS Supabase. | **Production** — serves `opencharts.org`. |
| `main` | The in-progress multi-tenant rewrite. Reads from a separate, OpenCharts-owned Supabase. | **Preview only** — lands at a preview URL like `main.opencharts.pages.dev`. |

The `production-ezra` → "production" mapping is set in the Cloudflare Pages project (**Settings → Builds & deployments → Production branch**). If that setting ever flips back to `main`, opencharts.org will start serving the in-progress rewrite — don't change it without a coordinated cutover.

Both branches auto-deploy via `.github/workflows/deploy.yml`. The workflow passes `--branch=${{ github.ref_name }}` so each git branch publishes to its own logical Pages branch, and CF decides which becomes production based on the setting above.

### Hotfixes
- Bug in the live site? Branch from `production-ezra`, PR back to `production-ezra`, merge. Auto-deploys to opencharts.org.
- Feature on the new platform? Branch from `main`, PR back to `main`, merge. Auto-deploys to preview only.

Don't merge `main` into `production-ezra` or vice versa — they're meant to diverge.

## Architecture (main branch / new build)

### Cloud Pieces
- **Cloudflare Pages** — hosts the SPA + Pages Functions for the API surface
- **OpenCharts Supabase** — a **separate** Supabase project from CMS's. Stores tenants, requests, events, notes, API keys, webhook secrets

### Repo layout
```
opencharts/
├── functions/api/                  # CF Pages Functions = the API
│   ├── _lib/                       # shared helpers (crypto, supabase, auth, responses, timeline)
│   ├── track/[hash].ts             # public read: GET /api/track/:hash
│   └── v1/                         # tenant API (Bearer auth)
│       ├── api-keys.ts             # POST + GET /api/v1/api-keys
│       └── requests/
│           ├── [id].ts             # GET /api/v1/requests/:id
│           └── [id]/events.ts      # POST /api/v1/requests/:id/events
├── src/                            # React SPA (Vite)
├── supabase/migrations/            # OpenCharts DB schema
└── scripts/create-tenant.ts        # bootstrap a tenant + first API key
```

### Auth model
- **Public timeline (`/api/track/:hash`)** — anon. The `public_tracking_hash` itself is the capability.
- **Tenant API (`/api/v1/*`)** — `Authorization: Bearer <token>` where the token is a 64-hex-char API key. Stored as SHA-256(token); plaintext shown exactly once at creation.
- **Webhook adapters (`/api/webhooks/*`)** — per-(tenant, service) shared secret in `X-OpenCharts-Webhook-Secret`. Not yet implemented as of PR #1.
- **Authoring UI (notes / incomplete flags)** — Supabase auth (magic link); `tenant_users` joins `auth.users` to a tenant. RLS scopes reads/writes by tenant. Not yet implemented as of PR #1.

### Token format
- 64 lowercase hex characters (256 bits of entropy)
- First 8 hex chars = `prefix` (displayed in UI, indexed for lookup)
- SHA-256(token) stored as `hashed_secret`
- Lookup is constant-time-equality verified after a prefix-indexed candidate fetch

### Public_tracking_hash format
- 22 URL-safe base64 chars (132 bits of entropy)
- Generated by a `BEFORE INSERT` trigger on `requests` so a hash always exists at the DB level
- Same shape as the CMS `providers.public_tracking_hash` so URL handling is identical

## Local development

```bash
# install once
npm install

# unit tests
npm test

# run the SPA
npm run dev

# run the SPA + CF Pages Functions together
# requires: .dev.vars (see .dev.vars.example) with Supabase URL + service-role key
npm run build && npm run pages:dev
```

The Functions need `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` at runtime — locally via `.dev.vars`, in production via the Cloudflare Pages project's environment variables (set both Production and Preview).

## Provisioning a Supabase project for OpenCharts

OpenCharts intentionally does **not** share a Supabase project with CMS. To stand up the new project:

1. Create a fresh Supabase project (suggested name: `opencharts-prod`).
2. Apply migrations:
   ```bash
   cd opencharts
   supabase link --project-ref <new-project-ref>
   supabase db push    # applies supabase/migrations/0001_init.sql
   ```
3. Grab `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_ANON_KEY` from the project's API settings.
4. Set them in Cloudflare Pages project env vars (Production + Preview).
5. Bootstrap the first tenant locally:
   ```bash
   SUPABASE_URL=... SUPABASE_SERVICE_ROLE_KEY=... \
     npm run tenant:create -- --name "Ezra Legal" --email "you@example.com"
   ```
   Capture the printed secret — it won't be shown again.

## Things to watch for

- **Don't put PHI in OpenCharts.** `patient_ref` on `requests` is an opaque correlation id supplied by the tenant; treat it as such. Anything tenants choose to put in note bodies is their responsibility, but the schema/API shouldn't encourage PHI in structured columns.
- **`public_tracking_hash` is a capability token.** Anyone with the hash can read the timeline. Don't accept writes keyed by hash alone — always require a service secret or tenant API key.
- **Service-role usage in Pages Functions is wide-open.** Every Pages Function that uses `getServiceSupabase` must handle auth before reading or writing. Lean on the helpers in `functions/api/_lib/`.
- **RLS exists for the future SPA path** (when the React app talks directly to Supabase via the anon key with a signed-in user). Today the Pages Functions bypass RLS via service-role, so RLS errors won't surface until the SPA layer lands.
